# Client Onboarding Playbook

Maturity: beta
Last updated: 2026-04-27

## Purpose

Use this as the current broad onboarding flow for Safeer's premium automation, AI-agent, internal-tool, and web-system projects.

Historical tool/niche-specific onboarding notes are preserved in `legacy-client-onboarding-playbook.md`.

---

## Good-fit clients

- founder-led businesses
- creators or service businesses with repeatable operations
- small teams with messy intake, follow-up, reporting, or handoffs
- ops-heavy teams that need cleaner visibility
- businesses that need a sharper website plus backend workflow cleanup

Avoid leading with a specific tool stack. Start with the workflow and outcome.

---

## Discovery checklist

### Business context

- What does the business sell?
- Who handles operations today?
- Where does work enter the system? Forms, DMs, calls, email, CRM, marketplace?
- What happens after a lead/order/request comes in?
- Where do delays or mistakes usually happen?

### Workflow mapping

- Trigger: what starts the workflow?
- Inputs: what data/files/messages are needed?
- Steps: what happens manually now?
- Decisions: where does a human need to approve or review?
- Outputs: what should be created, sent, updated, or reported?
- Owners: who needs visibility?

### System constraints

- Current website/platform
- Current forms/CRM/database/spreadsheets
- Messaging/email/calendar tools
- Reporting/dashboard needs
- Access/security constraints
- Budget and timeline

## Offer paths

| Offer | First useful version |
|---|---|
| Website / landing page build | clear offer page, CTA, contact path, analytics basics |
| Automation system | one workflow from intake to output with logging/alerts |
| AI assistant / internal tool | assistant for triage, drafting, routing, search, or reporting |
| Workflow teardown | mapped workflow, friction points, recommended build sequence |

---

## Proposal structure

1. Problem observed
2. Business outcome
3. First useful version
4. Scope boundaries
5. Timeline
6. Price/package
7. Access needed
8. Verification criteria
9. Handoff/support

---

## Definition of done

A project is not done until:

- the agreed workflow/page/tool works end-to-end
- edge cases are documented
- client can see where data goes
- errors/failures have a handling path
- handoff notes are delivered
- Safeer has a reusable proof asset if client details can be anonymized safely

---

## Proof capture

After each project, capture:

- before state
- after state
- what changed operationally
- time/frequency saved if defensible
- screenshots or diagrams with sensitive data removed
- client quote only with permission

Use proof for portfolio only after privacy/sensitivity review.
